## Using the Effect Hook(Effect Hooks 사용하기)
- Hooks 는 React 16.8 에서의 새로운 추가 사항입니다. Hooks 는 클래스를 작성하는것 없이 state와 React 의 다른 feature(특징)을 사용하게 합니다. 
~~~ 
    - Effect Hook은 function components 에서 side effects 를 실행하게 합니다.
~~~
~~~Java Script

import React, { useState, useEffect } from 'react';

function Example() {
  const [count, setCount] = useState(0);

  // Similar to componentDidMount and componentDidUpdate:
  //  componentDidMount 및 componentDidUpdate 와 유사합니다.
  useEffect(() => {
    // Update the document title using the browser API
    // 브라우저 API를 사용하여 document title을 업데이트 함.
    document.title = `You clicked ${count} times`;
  });

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>
        Click me
      </button>
    </div>
  );
}
~~~
    - 이 예제는 이전 페이지로부터의 counter 예제에 기반을 두고 있습니다만, counter 예제에 새로운 feature 을 추가하였습니다.
    우리는 document title을 사용자가 클릭한 숫자를 포함한 커스텀 메시지로 설정하였습니다.  
   
    - React 컴포넌트에서 data fetching, subscription 설정, 수동으로 DOM을 변경하기는 side effects의 모든 예시입니다. 
    "side effects" (or "effect")를  호출하는 이러한 동작에 당신이 익숙하든지 않든지, 아마도 당신의 컴포넌트에서 이전에 이러한 동작을
    실행하였을 것입니다.


Tip

    - 만약 React의 class 컴포넌트의 lifecycle 메서드에 익숙하다면, useEffect Hook를 componentDidMount, componentDidUpdate,그리고
    componentWillUnmount 가 합쳐진것으로 생각할 수 있습니다.

    - React 컴포넌트에는 두 가지의 흔한 종류의 side effect가 있는데, 하나는 cleanup을 필요로 하지 않고, 나머지는 필요로 합니다.
    이 차이를 더 자세하게 들여다 봅시다. 

## Effects Without Cleanup(cleanup 이 없는 Effects)
    - 때때로, 우리는 React가 DOM을 업데이트 한 후에 몇 추가적인 코드를 실행하고 싶습니다. Network requests, manual DOM mutations, and logging
    은 cleanup을 요구로 하지 않는 흔한 effects의 예제들입니다. 이렇게 이야기 할 수 있는 이유는 이러한 추가적인 코드를 실행하고, 즉시 코드를
    까먹어도 되기 때문입니다. 어떻게 class 컴포넌트와 Hooks 가 우리로 하여금 이러한 side effects를 표현하게 하는지 비교해 봅시다.



## Example Using Classes(class 컴포넌트를 이용한 예제)
    - class 컴포넌트를 이용한 React에서는, render method 자체는 side effects를 발생시키지 않습니다. 그 이유는 너무 일찍이여서 그런데 
    우리는 보통 우리의 effects를 React가 DOM을 업데이트 한 후에 실행하고 싶기때문입니다.

    - 그러한 이유로, React의 class 컴포넌트는 side effects를 componentDidMount and componentDidUpdate에서 실행되도록 하였습니다.
    다시 예제로 돌아와, 여기에 DOM에 변경사항을 만든 바로 후에 document title을 업데이트하는 React의 counter class 컴포넌트를 보여줍니다.

~~~Java Script
class Example extends React.Component {
  constructor(props) {
    super(props);
    this.state = {
      count: 0
    };
  }

  componentDidMount() {
    document.title = `You clicked ${this.state.count} times`;
  }
  componentDidUpdate() {
    document.title = `You clicked ${this.state.count} times`;
  }

  render() {
    return (
      <div>
        <p>You clicked {this.state.count} times</p>
        <button onClick={() => this.setState({ count: this.state.count + 1 })}>
          Click me
        </button>
      </div>
    );
  }
}
~~~

 **우리가 class 컴포넌트에서 어떻게 이러한 두 lifecycle 메서드들 사이의 코드를 중복해야하는지 주목해 주세요**

    - 이것은 많은 경우에 컴포넌트가 금방 마운트 됐는지, 업데이트가 되었는지의 여부와 관계없이 동일한 side effect를 수행하기를
    원하기 때문입니다. 개념적으로 우리는 side effect가 매번 렌더링을 할 때마다 일어나길 원합니다. 그러나 class 컴포넌트는 이러한 기능을
    가진 메서드를 가지고 있지 않습니다. 그래서 우리는 별개의 매서드를 추출해낼수 있지만, 여전히 두 개의 장소에서 메서드를 호출해야 합니다.
    (componentDidMount, componentDidUpdate에서와 같이)
   
    - 이제 useEffect Hook를 이용하여 어떻게 같은 일을 할 수 있는지 봅시다.




## Example Using Hooks(Hook를 이용한 예제)
    - 우리는 이미 이 페이지의 상단에서 이 예제를 보았지만, 좀 더 자세히 살펴 봅시다.
~~~Java Script

import React, { useState, useEffect } from 'react';

function Example() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `You clicked ${count} times`;
  });

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>
        Click me
      </button>
    </div>
  );
}
~~~
### What does useEffect do?
    - Hook를 이용함으로서, React에게 당신은 rendering 후에 당신의 컴포넌트가 무언가 할 필요가 있다고 말합니다. React는 당신이 전달한 함수를 기억할 것이며, 
    (우리는 전달한 함수를 우리의 "effect"로서 참조할겁니다.) DOM 업데이트를 수행한 후에, effect를 호출할 겁니다. 이 effect 에서, 우리는 document.title에 
    값을 설정하지만, 우리는 또한 data fetching 이나 몇 개의 다른 긴요한 API를 호출할 수 도 있습니다.)


### Why is useEffect called inside a component?
    - useEffect를 컴포넌트 내부에 위치시키는 것은 우리로 하여금 effect에서 바로state 변수인 count 변수에 접근하도록 합니다. 우리는 특별한 API로 state를
    읽기 위해 필요로 하지 않습니다. 왜냐면 state는 이미 함수 스코프에 위치하고 있기 때문이죠. Hooks는 JavaScript 클로저를 받아들이며, JavaScript가 
    이미 해결책을 제공하는곳에 React 특정 API를 도입하는 것을 지양합니다.

  
### Does useEffect run after every render?
    - 네! 기본설정으로, useEffect는 처음으로 렌더링할 때와 매 업데이트가 발생한 후 모두 후에 작동합니다. (나중에 어떻게 이것을 커스텀하는지에 대해 이야기할 겁니다.)
    "mounting"과 "updating"의 면에서 생각하는 대신에, effects 는 렌더링 후에 발생한다라고 생각하는것이 더 쉽다는 것을 알게 될겁니다. React는 컴포넌트가 effects를 실행할 
    쯤에 DOM이 업데이트 되었다는 것을 보장 합니다.


## Detailed Explanation(세부 설명)
    - effects에 대해 더 알게 되었으니 아래의 라인들이 이해 될겁니다.
~~~Java Script

function Example() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    document.title = `You clicked ${count} times`;
  });
}
~~~

    - 우리는 state 변수인 count를 선언한 후, React에세 effect를 사용할 필요가 있다고 말합니다. useEffect Hook에게 함수를 전달합니다.
    우리가 전달한 함수가 우리의 effect 입니다. 우리의 effect 내부에서, 우리는 document.title browser API를 사용하여 document title를 
    설정합니다. 우리는 effect 내부에서 가장 최근 업데이트 된 count 값을 읽을수 있는데, 그 이유는 effect 함수의 스코프내에 있기 때문입니다. 
    우리의 컴포넌트를 React가 렌더링 할 때, 우리가 사용한 effect를 기억하며, DOM을 업데이트 한 후에 우리의 effect를 실행합니다. 
    effect 실행은 제일 처음 렌더링을 포함한  매 렌더링 후에 발생합니다.
 
    - 숙련된 자바스크립트 개발자들은 useEffect로 전달되는 함수가 매 렌더링때마다 다를거라는 것을 눈치챌지 모릅니다. 이것은 고의 입니다. 
    사실 이것이 우리로하여금 state가 최신의 값이 아닌지에 대한 걱정 없이 effect 내부로부터 count 변수의 value 를 읽도록 하게 하는 것입니다.
    우리가 re-rendering을 할 때마다, 우리는 이전의 effect를 대체하는 다른 effect를 예약합니다. 어떤 면에서는, 이러한 동작이 effects 가 
    더 rendering의 결과의 한 부분으로서 동작하게 만듭니다. 각 effect는 특정한 rendering에 "소속"됩니다.  우리는 나중에 이 페이지에서 
    이게 왜 유용한지에 대해 더 명확히 살펴볼 것입니다. 

-----
Tip
- componentDidMount 및 componentDidUpdate과는 달리, useEffect 로 예정된 effects 는 브라우저가 스크린을 업데이트하는 것을 막지 않습니다. 이것이 당신의 어플리케이션으로 하여금 더 반응적으로 느끼게끔 만듭니다. effects의 대부분은 동기적으로 발생할 필요는 없습니다. layout을 측정하는 것과 같은 흔하지 않은 경우로, useEffect와 동일한 API를 가진 별개의useLayoutEffect Hook가 있습니다. 
-----

## Effects with Cleanup(cleanup 을 가진 Effects)
    - 일찍이, 어떠한 clean up도 요구하지 않는 side effects를 표현하는 방법을 살펴봤습니다.  그러나 어떤 effects 는 cleanup(메모리 회수 인듯)을 요구로 합니다. 예를 들어 우리는 어떤 외부 데이터 소스에 대한 구독을 설정하길 원할지도 모릅니다. 이러한 경우에 우리는 어떠한 메모리 누수도 발생하는 것을 원치 않기때문에, clean up을 하는 것이 중요합니다. class 컴포넌트와 Hooks 를 가진 function 컴포넌트를 비교하면서 어떻게 메모리 회수를 할 수 있는지 봅시다.


## Example Using Classes(클래스를 사용하는 예시)
    - React 클래스에서, 대개 componentDidMount에서 subscription을 설정하며, componentWillUnmount에서 subscription을 회수합니다. 예를 들어 우리가 친구의 온라인 상태를 구독하게끔하는 ChatAPI 모듈을 가지고 있다고 합시다. 여기에서 어떻게 우리가 subscribe하는지 그리고  클래스를 사용하여 status를 보여줍니다.

~~~Java Script
class FriendStatus extends React.Component {
  constructor(props) {
    super(props);
    this.state = { isOnline: null };
    this.handleStatusChange = this.handleStatusChange.bind(this);
  }

  componentDidMount() {
    ChatAPI.subscribeToFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }
  componentWillUnmount() {
    ChatAPI.unsubscribeFromFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }
  handleStatusChange(status) {
    this.setState({
      isOnline: status.isOnline
    });
  }

  render() {
    if (this.state.isOnline === null) {
      return 'Loading...';
    }
    return this.state.isOnline ? 'Online' : 'Offline';
  }
}
~~~
     - 어떻게 componentDidMount 와 componentWillUnmount가 서로의 모습을 보여줄 필요가 있는지 주목하세요. 
     (componentDidMount and componentWillUnmount).
     lifecyle 메서드는 비록 개념적으로 두 메서드에 있는 코드가 같은 effect와 관계가 있을지라도 이러한 로직을 분리하도록 강요합니다. 

-----
Note

눈매가 매서운 독자들은 또한 어쩌면 이 예제가 완전히 정확하기 위해서 componentDidUpdate를 필요로 한다는 것을 눈치 챌겁니다. 현재로서는 
componentDidUpdate가 없다는 것을 무시하지만 이 페이지의 뒤의 부분에서 다시 살펴 볼 것입니다.

-----

## Example Using Hooks(Hooks를 이용한 예제_
    - Hooks를 가지고 위의 컴포넌트를 어떻게 작성할 수 있는지 봅시다.

    - 어쩌면 당신은 cleanup을 수행하기위해 별도의 effect가 필요하다고 생각할지 모르지만, subscription을 추가하고, 제거하는 코드는 단단히 연결되어 있기때문에, useEffect는 두 역할을 함께 묶도록 디자인되어 있습니다. 만약 당신의 effect가 함수를 반환한다면, React는 cleanup을 할때가 오면 실행할 것입니다.

~~~Java Script

import React, { useState, useEffect } from 'react';

function FriendStatus(props) {
  const [isOnline, setIsOnline] = useState(null);

  useEffect(() => {
    function handleStatusChange(status) {
      setIsOnline(status.isOnline);
    }
    ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);
    // Specify how to clean up after this effect:
    // 이 effect 이후에 clean up 하는 법을 명세화하세요.
    return function cleanup() {
      ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange); // 여기 주목!
    };
  });

  if (isOnline === null) {
    return 'Loading...';
  }
  return isOnline ? 'Online' : 'Offline';
}
~~~

### Why did we return a function from our effect? (왜 우리의 effect 후에 함수를 반환했는가?
    - 이것은 effects를 위한 선택적인 cleanup 메커니즘 입니다. 매 effect는 effect 함수를 수행한 후에 clean up하는 함수를 실행합니다.이것이 우리로 하여금 subscriptions을 추가하고 제거하는 로직이 서로 밀접하게 연결되어 있도록 유지합니다. 이러한 함수들은 같은 effect내의 부분입니다.!

### When exactly does React clean up an effect? (React는 언제 정확히 effect를 청소하나요?)
    - React는 컴포넌트가 unmount 할 때 cleanup을 수행합니다. 그러나 우리가 이전에 배운대로, effects는 단 한번이 아니라 매 렌더링때마다 작동합니다. 이것이 바로 React가 다음번에 effect를 실행하기전에 이전의 렌더링으로부터의 effects를 cleanup 하는 이유입니다. 우리는 왜 이것이 버그를 예방하도록 돕는지, 성능이슈가 발생하는 경우 이 성능저하에서 어떻게 빠져 나오는지 아래에서 이야기를 나눌겁니다.
----
Note
- 우리는 effect로부터 이름이 있는 함수를 반환할 필요가 없습니다. cleanup()이라고 이름 지은 이유는 그것의 목적을 명확히보여주기 위해서입니다. 그러나 당신은 화살표 함수 () => {} 를 반환하거나 다른 이름으로 호출이 가능합니다. 
----

## Recap (복습)
    - 우리는 useEffect가 우리로 하여금 컴포넌트가 렌더링 한 후에 다른 종류의 side effects 를 표현하게끔 한다는 것을 배웠습니다. 어떤 effects는 cleanup을 필요로 할수 잇어 우리는 함수를 반환합니다.
~~~Java Script
  useEffect(() => {
    function handleStatusChange(status) {
      setIsOnline(status.isOnline);
    }

    ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);
    return () => {
      ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange);
    };
  });
  
~~~
    - 다른 effects는 cleanup 단계를 가지지않고, 아무것도 반환하지 않습니다. 
~~~Java Script

  useEffect(() => {
    document.title = `You clicked ${count} times`;
  });
~~~

    - Effect Hook은 단 하나의 API로 위의 두 가지 사용 케이스들을 통합합니다.


** 만약 어떻게 Effect Hook가 작동하는지에 대해 잘 파악했다고 생각하거나 또는 많은 정보로 압도되었다면, [Rules of Hooks](https://reactjs.org/docs/hooks-rules.html)로 지금 넘어가도 좋습니다.**

## Tips for Using Effects(Effects를 사용하는 팁)
    - 우리는 숙련된 React 사용자들이 아마도 궁금해 할만한 useEffect의 몇가지 측면을 더 심도깊은 관점으로 이 페이지를 계속해 나갈겁니다. Effect Hooks를 지금 깊게 공부해야한다는 의무감을 느끼지 마세요. Effect Hook에 대한 더 많은 세부사항을 배우기 위해 언제든지 이 페이지에 돌아 올수 있습니다.


### Tip: Use Multiple Effects to Separate Concerns (Tips: 개별적인 관심사에 대한 여러개의 Effects를 사용하세요.)
    - Hooks에 대해서 앞의 챕터인 `Motivation`에서 우리가 간략히 말한 문제 중의 하나는 class 컴포넌트의 lifecyle 메서드는 종종 서로 관계가 없는 로직을 포함하고 있다는 겁니다. 그러나 관계있는 로직은 여러 개의 메서드로 나누어져 있습니다. 여기에 이전 예로부터의 counter와 friend status indicator 로직을 합치는 컴포넌트가 있습니다.

~~~Java Script
class FriendStatusWithCounter extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0, isOnline: null };
    this.handleStatusChange = this.handleStatusChange.bind(this);
  }

  componentDidMount() {
    document.title = `You clicked ${this.state.count} times`;
    ChatAPI.subscribeToFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }

  componentDidUpdate() {
    document.title = `You clicked ${this.state.count} times`;
  }

  componentWillUnmount() {
    ChatAPI.unsubscribeFromFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }

  handleStatusChange(status) {
    this.setState({
      isOnline: status.isOnline
    });
  }
  // ...
 ~~~
     - 어떻게 document.title 값을 대입하는 로직이 componentDidMount와 componentDidUpdate 사이에 분리되어 있는지 주목하세요. subscription 로직은 또한 componentDidMount과 componentWillUnmount에 퍼져 잇습니다. 그리고 componentDidMount는 두 작업에 대한 코드를 포함하고 있습니다. 

    - 그리하여, Hooks는 이 문제를 어떻게 해결할 수 있나요? State Hook를 한 번 이상 사용하는 것과 같이, 당신은 또한 여러 개의 effects를 사용할 수 있습니다. 이것이 우리로 하여금  관계 없는 로직이 다른 effects로 분리하게 합니다.

~~~Java Script
function FriendStatusWithCounter(props) {
  const [count, setCount] = useState(0);
  useEffect(() => {
    document.title = `You clicked ${count} times`;
  });

  const [isOnline, setIsOnline] = useState(null);
  useEffect(() => {
    function handleStatusChange(status) {
      setIsOnline(status.isOnline);
    }

    ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);
    return () => {
      ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange);
    };
  });
  // ...
}
~~~

    - Hooks는 lifecyle 메서드 이름 보다는 무엇을 하고 있는가에 기준을 두어 코드를 분리하게 합니다. React는 명시된 순서대로 컴포넌트에 의해 사용되는 모든 effect를 적용할겁니다.



## Explanation: Why Effects Run on Each Update ( 풀이 : 왜 Effects 가 업데이트할 때마다 실행되는가)
    - 만약 class 컴포넌트에 적응되었다면 당신은 왜 effect의 cleanup 단계가 리-렌더링할때마다 발생하는지 궁금할 겁니다. 언마운팅하는 동안의 딱 한번만이 아니라요.
    왜 이 디자인이 우리로 하여금 더 적은 버그들을 가진 컴포넌트들을 생성하도록 도와주는지 실용적인 예제를 살펴봅시다.


    - 이 페이지의 앞에서, 우리는 친구가 온라인인지 아닌지를 보여주는 FreiendStatus 컴포넌트라는 예시로 시작하였습니다.우리의 class 컴포넌트는 friend.id 를 this.props 에서 읽어들이며, 컴포넌트가 마운트한 후에 friend status 를 구독합니다. 그리고 언마운팅 하는 동안 구독을 해제합니다.
~~~Java Script
  componentDidMount() {
    ChatAPI.subscribeToFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }

  componentWillUnmount() {
    ChatAPI.unsubscribeFromFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }
  ~~~
  
    - 그러나 만약 컴포넌트가 스크린에 보이는 동안 friend prop이 변한다면 무슨일이 발생하나요? 우리의 컴포넌트는 다른 친구의 온라인 상태를 계속해서 보여줄겁니다. 이것은 `버그` 입니다. 우리는 또한 unsubscribe 호출이 잘못된 friend ID를 사용할 것이기에 unmounting 시 메모리 누수나 crash를 야기할 겁니다. 

    - class 컴포넌트에서, 우리는 이 경우를 처리하기 위해 componentDidUpdate를 추가할 것을 필요로 할겁니다.

~~~Java Script

  componentDidMount() {
    ChatAPI.subscribeToFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }

  componentDidUpdate(prevProps) {
    // Unsubscribe from the previous friend.id
    // 이전의 friend.id 로부터 unsubscribe 하기
    ChatAPI.unsubscribeFromFriendStatus(
      prevProps.friend.id,
      this.handleStatusChange
    );
    // Subscribe to the next friend.id
    // 다음의 friend.id로 subscribe 하기
    ChatAPI.subscribeToFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }

  componentWillUnmount() {
    ChatAPI.unsubscribeFromFriendStatus(
      this.props.friend.id,
      this.handleStatusChange
    );
  }
 ~~~
 
    - 적절히 componentDidUpdate를 처리하는 것을 까먹는것은 React 어플리케이션에서 흔한 버그 원인 입니다.
    - 이제 Hooks를 사용하는 이 컴포넌트의 버전을 생각해보세요.

~~~Java Script

function FriendStatus(props) {
  // ...
  useEffect(() => {
    // ...
    ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);
    return () => {
      ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange);
    };
  });
 ~~~
    - Hooks를 사용한 클래스는 버그로부터 고통받지 않습니다.(그러나 이 클래스에 어떠한 것도 변경하지 않았습니다.)
    
    - useEffect는 기본설정으로 업데이트를 처리하기때문에, 업데이트를 처리하는 특별한 코드가 없습니다. 다음의 effects 를 적용하기전에 이전의 effects를 청소합니다. 
    이것을 설명하기위에 여기에 이 컴포넌트가 지속적으로 생성할  연속적인 subscribe 와 unsubscribe 호출을 아래에 보여줍니다.
    

~~~Java Script
// Mount with { friend: { id: 100 } } props // { friend: { id: 100 } } props 으로 마운트
ChatAPI.subscribeToFriendStatus(100, handleStatusChange);     // Run first effect // effect를 처음으로 실행

// Update with { friend: { id: 200 } } props // { friend: { id: 200 } } props 으로 업데이트
ChatAPI.unsubscribeFromFriendStatus(100, handleStatusChange); // Clean up previous effect // 이전 effect를 청소
ChatAPI.subscribeToFriendStatus(200, handleStatusChange);     // Run next effect // 다음 effect를 실행

// Update with { friend: { id: 300 } } props // { friend: { id: 300 } } props 으로 업데이트
ChatAPI.unsubscribeFromFriendStatus(200, handleStatusChange); // Clean up previous effect  // 이전 effect를 청소
ChatAPI.subscribeToFriendStatus(300, handleStatusChange);     // Run next effect // 다음 effect를 실행

// Unmount // 언마운트 
ChatAPI.unsubscribeFromFriendStatus(300, handleStatusChange); // Clean up last effect // 마지막 effect를 청소
~~~

    - 이 동작들이 기본설정으로 일관성을 보장하며, update 로직을 놓쳐서 생기는 class 컴포넌트에선 흔한 버그를들 예방합니다.


## Tip: Optimizing Performance by Skipping Effects(Effects를 건너뜀으로서 성능을 최적화하기)
    - 어떤 경우에는, 매 렌더링 후에 effect를 적용하거나 cleaning up 하는 것이 성능 문제를 생성할 수 있습니다. class 컴포넌트에서는 componentDidUpdate내부에 prevProps 나 prevState로  추가적인 비교작업을 작성함으로써 이문제를 해결할 수 있습니다. 

~~~Java Script
componentDidUpdate(prevProps, prevState) {
  if (prevState.count !== this.state.count) {
    document.title = `You clicked ${this.state.count} times`;
  }
}
~~~

    - 이러한 자격요건은 충분히 흔해서 useEffect Hook API로 내장되었습니다. 당신은 React에게 만약 특정 values가 리-렌더링 사이에 변하지 않았다면 effect를 적용하는 것을 건너뛰라고 말할 수 있습니다. 그렇게 하기위해 선택적인 두번째 인자로서 배열을 useEffect로 전달합니다.
~~~Java Script

useEffect(() => {
  document.title = `You clicked ${count} times`;
}, [count]); // Only re-run the effect if count changes // 오직 count가 변할때만 effect를 재 실행합니다.
~~~
    - 위의 예제에서, 우리는 [count]를 두번째 인자로서 전달합니다. 이것이 무엇을 의미하나요? 만약 count가 5이고, 그리고 나서 우리의 컴포넌트가 재 렌더링을 할때 count가 여전히 5 이라면, React는 이전의 렌더링으로 부터의 [5]와 다음의 렌더링의 [5]와 비교합니다. 배열에서의 모든 items 이 같기 때문에 (5 === 5), React는 effect를 건너뛸겁니다. 이것이 우리가 말하는 최적화입니다.
    - 우리가 값이 6으로 업데이트된 count로 렌더링할때, React는 이전의 렌더링으로부터의 배열 [5]와 다음 렌더링으로부터의 배열 [6]에 있는 items 를 비교합니다. 이번에는 React는 effect를 재적용할터인데 그 이유는  5 !== 6 이기 때문입니다. 만약 array에 여러개의 items가 있고, React는 그 중하나라도 다르다면 effect를 재 실행할 겁니다.

    - 또한  cleanup 단계를 가지는 effects 에게도 작동합니다.


~~~Java Script
useEffect(() => {
  function handleStatusChange(status) {
    setIsOnline(status.isOnline);
  }

  ChatAPI.subscribeToFriendStatus(props.friend.id, handleStatusChange);
  return () => {
    ChatAPI.unsubscribeFromFriendStatus(props.friend.id, handleStatusChange);
  };
}, [props.friend.id]); // Only re-subscribe if props.friend.id changes  // 만약 props.friend.id가 변한다면, re-subscribe하라 // 여기 주목!!


~~~
    - 미래에 두번째 인자는 빌드타임 변경에 의해 자동적으로 추가될지도 모릅니다.
In the future, the second argument might get added automatically by a build-time transformation.

Note

- 만약 이 최적화를 당신이 사용한다면, 시간이 지남에 따라 변하거나 effect에 의해 사용되는 컴포넌트 scope로의 모든 값을 배열이 포함한다는 것을 확실히 하세요(props와 state와 같은). 그렇지 않으면 당신의 코드는 이전의 렌더링으로부터 업데이트 되지 않은 values을 참조할 겁니다. 함수를 어떻게 다룰지, 배열이 너무 자주 변경될때 무엇을 할지를 더 배워주세요.
If you use this optimization, make sure the array includes all values from the component scope (such as props and state) that change over time and that are used by the 

- 만약 effect를 실행하고, clean up 오직 한번 씩 각각 실행한다면, (mount와 unmount 시), 당신은 빈 배열([])을 두번째 인자로서 전달할 수 있습니다. 이것은 React에게 당신의 effect가props나 state로부터의 어떤 values들에도 의존하고 있지 않다라는 것을 말해줍니다. 그래서 effect를 재 실행할 필요가 전혀없습니다.  재실행하지 않는것이 특별한 경우로 취급되지는 않고, 어떻게 dependencies 배열이 항상 작동하는지 로부터 직접적으로 따라갑니다.

- 만약 빈 배열([])을 전달한다면 effect 내부에 있는 props와 state는 항상 초기화 값을 가질겁니다. []을 두번째인자로서 전달하는 것이 익숙한 componentDidMount 와 componentDidMount, componentWillUnmount 심성모델로 더 가까워지는 동안 effects를 너무 자주 리렌더링하는 것을 피하는 더 나은 해결책들이 보통 있습니다. 또한 React는 브라우저가 그려진 후 까지 useEffect를 실행하는 것을 미룬다는 것을 잊지마세요. 그래서 추가적인 작업을 하는 것은 별로 문제가 아닙니다. 

- 우리의 eslint-plugin-react-hooks package의 부분으로서  exhaustive-deps 규칙을 사용할 것을 권장합니다.  exhaustive-deps 규칙은 dependencies가 잘못되게 명시되어 있을때 경고하고 수정사항을 제시합니다.
